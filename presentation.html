The Problem

Extracting services from a monolith is the riskiest refactoring category. Three failure modes kill most attempts:

Cutting without pinning

You start the extraction before locking current behaviour. Regressions appear weeks later in production.

Leaking the old model

The new service exposes fields like lastPlayedBy and playCount. You extracted code but not the coupling.

No rollback plan

The 3am cutover has no decision tree. A failing health check becomes a 9am incident review.

The Approach

Strangler fig with test-pinned extraction. Every extraction follows the same gated sequence:

pin behaviour
→
rank seams
→
extract
→
fence boundary
→
write runbook
→
score quality

Claude Code coordinates the work across 6 phases. No phase starts until the previous gate is verified by running tests — not by Claude's judgement, by the test suite.

Alongside the backend extraction, the original AngularJS 1.2.16 SPA (2013, EOL) was replaced with a modern Vue 3 + Vite + Tailwind CSS frontend. All original features were preserved — the Spring Boot REST API was not touched.

Key frontend decisions

Vue 3 over React / Angular

Less boilerplate than React for a CRUD app this size. Angular 17+ would have been a near-complete rewrite with heavy tooling. Vue's Composition API maps cleanly onto AngularJS controller patterns.

Tailwind over Bootstrap

Drops the jQuery dependency and the outdated Bootstrap 3 icon sprites entirely. Utility classes replace both app.css and multi-columns-row.css with no custom stylesheet.

Decoupled build (dist/ not in Gradle)

npm run build outputs to frontend/dist/. No Java install required for frontend dev. When Java is available, one Gradle exec task and a build.outDir change wire it into the JAR.

Plain JS over TypeScript

Keeps the scaffolding lightweight for a demo app. The API layer and composables are small enough that type inference covers most cases; TypeScript can be layered on without restructuring.

How Claude Was Taught to Work Here

Three-level context hierarchy

A single project-level instruction like "don't add to the monolith" is easy for an LLM to ignore when it's deep in a file. A directory-level CLAUDE.md that Claude reads the moment it opens any file in music/ is impossible to miss.

LevelFileWhat it enforces
User~/.claude/CLAUDE.mdEnglish, conventional commits, terse style — applies to all projects
ProjectCLAUDE.mdService boundary rules, agent order, test tag conventions
Directorysrc/.../music/CLAUDE.md"You are in the legacy monolith root. Prefer extraction over addition."
Directorynew-service/CLAUDE.md"Never reference monolith packages. These field names are forbidden in the API."

Hooks for hard rules, prompts for preferences

Two different mechanisms enforce two different things:

PreToolUse Hook (Phase 5)

Blocks any Edit or Write that puts lastPlayedBy, playCount, recommendedFor, or _class into catalog-service/. Exits 1 — the edit never happens. Cannot be reasoned around.

Directory CLAUDE.md (preference)

"Prefer extraction over adding here" — guidance with legitimate exceptions. A bug fix in the monolith while extraction is in progress is valid. The hook doesn't block that.

Parallel agent spawning with explicit context

Three of the six phases spawn two specialised agents simultaneously in a single coordinator message. Each subagent receives its full context explicitly in the prompt — package paths, class lists, dependency graphs, scoring rubrics. Subagents do not inherit the coordinator's conversation. This makes each agent independently reproducible and prevents context bleed between concerns.

Custom Slash Commands

Reusable playbooks in .claude/commands/ — callable by any engineer, not just the person who set up the project:

  • /extract-service <seam-name> 7-step extraction playbook: verify characterization suite → locate seam → create module → strangler proxy → ACL translator → verify both suites green → report what couldn't be cut cleanly
  • /pin-behavior Runs ./gradlew test --tests "*.characterization.*", produces a pass/fail table, and recommends whether each failure is a regression or an intentional change — without ever modifying a pin without explicit instruction
  • /score-seam <seam-name> Gathers package contents and dependency graph, spawns a Task subagent with a structured scoring rubric (coupling 1–5, test coverage %, data-model tangle 1–5, business criticality), compares the verdict against the human ADR ranking

Key Design Insights

  • → Characterization tests pin field names, not just values — so a rename breaks the test with a message that says exactly what changed, not just "something broke"
  • → The strangler proxy defaults to false — monolith serves all traffic until ops explicitly flips the flag per environment. Zero risk on deploy.
  • → The eval harness (Phase 6) tracks false-confidence rate — the % of wrong proposals where Claude reported confidence ≥ 0.8. A model that knows what it doesn't know is more useful than one that is wrong confidently.
  • → The agentic-scouts output is compared against the human ADR ranking. Agreements validate the ADR; disagreements expose hidden risk the human author missed.
  • → The cutover runbook is generated before cutover, rehearsed once, and includes a Mermaid decision tree — so the 3am engineer makes zero judgement calls.
  • → Planning mode caught the Java/Gradle constraint before any code was written — the constraint surfaced in requirements, not in implementation, saving a full rework of the build integration.
  • → The 1-to-1 mapping from AngularJS controllers and directives to Vue 3 components was scaffolded in minutes; understanding AngularJS 1.x internals well enough to do this manually would have taken hours of reading dated documentation.
